3 Architecture Approaches

Each handles Claude Code + Desktop differently. Pick the one that fits.

A

Hook-Only (Claude Code)

A stop hook fires every time Claude finishes responding. The hook script strips code/tool output, calls ElevenLabs, plays audio. Slash commands toggle it on/off and pick the voice.

Pros

  • Fully automatic
  • Zero Claude involvement
  • Simple setup

Cons

  • Claude Code only
  • Desktop needs separate solution later
B

Hook + MCP Server ★ Recommended

Hook handles Claude Code automatically. An MCP server exposes a speak() tool — Claude Desktop uses a custom system prompt to call it at the end of each response.

Pros

  • Works on both surfaces
  • One TTS engine, two entry points
  • Voice/toggle control via MCP tools too

Cons

  • Desktop needs a system prompt
  • Slightly more setup
C

Standalone TTS Daemon

A local HTTP server (always running) handles all TTS. Hook and MCP both talk to it. Future: browser extension could hit the same endpoint for claude.ai web.

Pros

  • Most extensible
  • One place to manage queue, state, voices
  • Web support possible later

Cons

  • Always-on process to manage
  • Overkill for now

My recommendation

Option B — the hook handles Claude Code automatically (no effort on your part), and the MCP server gives Desktop a clean integration path. You can skip the Desktop system prompt for now and add it later. It's the right shape without being over-engineered.
